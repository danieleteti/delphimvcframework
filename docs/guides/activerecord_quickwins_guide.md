# TMVCActiveRecord Quick Wins Guide

Three additive features added to `TMVCActiveRecord`. All opt-in ‚Äî existing
classes are unaffected.

1. **Dirty Tracking** ‚Äî UPDATE only the columns that actually changed.
2. **`foRefresh` field option** ‚Äî refetch server-set values (DB DEFAULT,
   computed columns, triggers) after INSERT/UPDATE.
3. **Soft Delete** ‚Äî turn `Delete` into a marker UPDATE; SELECT auto-filters
   soft-deleted rows.

## Dirty Tracking

### Why

A naive `Update` writes every Updatable column, even those the user did
not change. For wide tables, this means more bytes on the wire, more
trigger firing, more contention with concurrent transactions. Dirty
tracking emits SET only for the fields whose value actually differs
from the last persisted state.

### Enable

```pascal
[MVCTable('orders')]
[MVCDirtyTracking]                // class-level, opt-in
TOrder = class(TMVCActiveRecord)
  ...
end;
```

### Use

```pascal
lOrder := TMVCActiveRecord.GetByPK<TOrder>(42);
try
  lOrder.Total := 99.99;
  Assert(lOrder.IsDirty);
  Assert(lOrder.IsDirty('total'));
  Assert(not lOrder.IsDirty('notes'));
  lOrder.Update;                  // emits: UPDATE orders SET total = ?, version = version + 1 WHERE id = ? AND version = ?
  Assert(not lOrder.IsDirty);     // snapshot rehashed in-memory after Update
finally
  lOrder.Free;
end;
```

### Edge cases

- Calling `Update` with **zero dirty fields** still emits SQL: SET
  becomes `pk = pk` (no-op data change). The statement runs so triggers
  fire, version increments where applicable, and `OnAfterUpdate`
  invokes. This intentional "lock-touch" behaviour matches the SQL
  conventional use of `UPDATE` to acquire a row-level write lock.
- Use `UpdateIfDirty` to opt out of that behaviour when you want a
  silent no-op:
  ```pascal
  lOrder.UpdateIfDirty;  // skips SQL entirely if nothing dirty
  ```
- `UpdateIfDirty` on a class without `[MVCDirtyTracking]` raises
  `EMVCActiveRecord` ("dirty tracking not enabled").
- Audit columns (`MVCAuditUpdatedAt` / `MVCAuditUpdatedBy`) are always
  included in the SET when their values change in `OnBeforeUpdate`.

### Memory cost

Hash-based snapshot. Per instance: `4 bytes √ó N tracked fields`. For 20
tracked fields = 80 bytes per instance. Independent of value sizes
(strings, blobs).

For instances of classes WITHOUT `[MVCDirtyTracking]`, the snapshot is
not allocated ‚Äî zero memory cost.

### Collision risk

CRC32 is not cryptographic. The chance that two different values for
the same field hash to the same Cardinal is approximately 1 in 4
billion. For business data this is negligible. The cost of a collision
is a false-negative: the field is considered unchanged and not
included in the UPDATE ‚Äî a missed write. If your data domain has
adversarial inputs (e.g. user-supplied values that could be crafted to
collide), do not rely on dirty tracking for correctness; use
unconditional `Update` instead.

## `foRefresh` field option

### Why

Some columns are populated by the database, not by the application:
- Server-side defaults (`created_at TIMESTAMP DEFAULT NOW()`).
- Computed columns (`full_name AS first_name || ' ' || last_name`).
- Triggers that modify the row on INSERT or UPDATE.

After persisting, the in-memory instance must be updated with the
DB-assigned values. `foRefresh` makes this declarative.

### Enable

```pascal
TOrder = class(TMVCActiveRecord)
  [MVCTableField('id', [foPrimaryKey, foAutoGenerated])]   FID: Int64;
  [MVCTableField('total')]                                  FTotal: Currency;
  [MVCTableField('created_at', [foRefresh])]                FCreatedAt: TDateTime;  // DB DEFAULT NOW()
  [MVCTableField('row_version', [foRefresh])]               FRowVersion: Int64;     // DB trigger updates
end;
```

### Use

Just call `Insert` or `Update` as usual. The framework refetches the
foRefresh fields automatically.

```pascal
lOrder := TOrder.Create;
try
  lOrder.Total := 100.0;
  lOrder.Insert;
  // lOrder.FCreatedAt is now populated with the DB-assigned timestamp,
  // not the Delphi default of 0.
finally
  lOrder.Free;
end;
```

### Per-engine implementation

Whenever supported, the framework uses a single statement with
`RETURNING` or `OUTPUT`. Only MySQL and MariaDB UPDATE require a
separate fallback `SELECT`.

| Engine | INSERT | UPDATE |
|---|---|---|
| PostgreSQL | `INSERT ‚Ä¶ RETURNING ‚Ä¶` | `UPDATE ‚Ä¶ RETURNING <foRefresh cols>` |
| Firebird | `INSERT ‚Ä¶ RETURNING ‚Ä¶` | `UPDATE ‚Ä¶ RETURNING <foRefresh cols>` |
| SQLite (3.35+) | Hybrid: legacy `last_insert_rowid()` when only PK; `RETURNING` when `foRefresh` is present | `UPDATE ‚Ä¶ RETURNING <foRefresh cols>` |
| Oracle (12c+) | `INSERT ‚Ä¶ RETURNING <cols> INTO :params` | `UPDATE ‚Ä¶ RETURNING <cols> INTO :params` |
| MSSQL | `INSERT ‚Ä¶ OUTPUT inserted.<cols>` | `UPDATE ‚Ä¶ SET ‚Ä¶ OUTPUT inserted.<cols> WHERE ‚Ä¶` |
| MySQL | `INSERT ‚Ä¶; SELECT LAST_INSERT_ID() AS pk` then separate `SELECT <cols> WHERE pk=?` | Same fallback `SELECT` after UPDATE |
| MariaDB (10.5+) | `INSERT ‚Ä¶ RETURNING ‚Ä¶` | Inherits MySQL fallback (MariaDB has no UPDATE RETURNING) |

### MariaDB vs MySQL

The framework ships separate SQL generators for MySQL and MariaDB.
Drop-in compatibility is one-way:

- Code written for MySQL works on MariaDB (MariaDB executes all MySQL
  SQL).
- Code written for MariaDB does NOT necessarily work on MySQL
  (MariaDB-specific syntax fails on MySQL).

To opt into MariaDB optimizations, register the connection with the
backend tag `'mariadb'`:

```pascal
ActiveRecordConnectionsRegistry.AddDefaultConnection(lConn, 'mariadb');
```

### SQLite caveat

`RETURNING` requires SQLite 3.35.0 (March 2021) or newer. FireDAC
bundles a recent SQLite engine ‚Äî but the bundled DLL on some Delphi
versions may be older. The framework uses a **hybrid strategy** for
SQLite to stay compatible:

- When only `foAutoGenerated` is present (no `foRefresh`): legacy
  `last_insert_rowid()` workaround. Works on any SQLite version.
- When `foRefresh` is present: `RETURNING`. Requires SQLite 3.35+.

If you hit a runtime error on `RETURNING`, replace the bundled
`sqlite3.dll` with a recent one from sqlite.org.

## Soft Delete

### Why

Audit, retention, and recovery scenarios require keeping deleted rows
in the database with a marker rather than physically removing them.
Compliance regimes (GDPR right-to-erasure, financial record
retention) drive much of the demand.

### Enable

Field-level attribute. Field type drives the mode:

```pascal
// Timestamp mode (recommended)
[MVCTable('customers')]
TCustomer = class(TMVCActiveRecord)
  [MVCTableField('id', [foPrimaryKey, foAutoGenerated])] FID: Int64;
  [MVCTableField('name')] FName: string;
  [MVCTableField('deleted_at')]
  [MVCSoftDeleted]
  FDeletedAt: NullableTDateTime;  // <-- MUST be NullableTDateTime, NOT plain TDateTime
end;

// Flag mode (for legacy schemas)
[MVCTable('orders')]
TOrder = class(TMVCActiveRecord)
  [MVCTableField('id', [foPrimaryKey, foAutoGenerated])] FID: Int64;
  [MVCTableField('description')] FDescription: string;
  [MVCTableField('is_deleted')]
  [MVCSoftDeleted]
  FIsDeleted: Boolean;
end;
```

**Important ‚Äî timestamp mode REQUIRES `NullableTDateTime`.** A plain
`TDateTime` defaults to `0.0`, which serializes to the date
`1899-12-30` (not NULL). The auto-filter `WHERE deleted_at IS NULL`
would never match ‚Üí all rows always look "deleted". Use the nullable
wrapper.

### Use

#### Soft-delete an instance

```pascal
lCustomer.Delete;
// Equivalent to: UPDATE customers SET deleted_at = Now() WHERE id = ? AND version = ?
```

For flag mode: `SET is_deleted = TRUE` (or `= 1` on MSSQL).

#### Hard-delete an instance

```pascal
lCustomer.HardDelete;
// Always physical: DELETE FROM customers WHERE id = ? AND version = ?
```

Use this for GDPR right-to-erasure or retention cleanup. Bypasses the
soft-delete attribute entirely.

#### Restore an instance

```pascal
lCustomer.Restore;
// UPDATE customers SET deleted_at = NULL WHERE id = ? AND version = ?
```

Raises `EMVCActiveRecord` if the class lacks `[MVCSoftDeleted]` or if
the record is currently alive (`IsDeleted = False`).

#### Check soft-delete state

```pascal
if lCustomer.IsDeleted then ...
```

Returns `False` for classes without `[MVCSoftDeleted]` (graceful
default ‚Äî calling `IsDeleted` is always safe).

#### Auto-filter SELECT

By default, every SELECT against a `[MVCSoftDeleted]` class
auto-injects a `WHERE deleted_at IS NULL` (or `WHERE is_deleted =
FALSE` for flag mode):

```pascal
lAlive := TMVCActiveRecord.All<TCustomer>;
// Returns only rows where deleted_at IS NULL
```

To see soft-deleted rows (audit screen, restore UI, GC):

```pascal
TMVCActiveRecord.IncludeSoftDeleted(True);
try
  lAll := TMVCActiveRecord.All<TCustomer>;
  // Returns ALL rows, alive and deleted
finally
  TMVCActiveRecord.IncludeSoftDeleted(False);
end;
```

The `IncludeSoftDeleted` flag is **thread-local** ‚Äî safe in
multi-threaded servers as long as the try/finally is observed.

#### Bulk operations

```pascal
TMVCActiveRecord.DeleteRQL<TCustomer>('le(score, 0)');
// Smart: soft-delete classes get UPDATE; non-soft classes get physical DELETE.

TMVCActiveRecord.HardDeleteRQL<TCustomer>('le(score, 0)');
// Always physical DELETE, even for soft classes.

TMVCActiveRecord.RestoreRQL<TCustomer>('eq(name, "Alice")');
// Bulk UPDATE deleted_at = NULL.
```

### Known limitations

- `SelectRQL<T>` does NOT currently auto-filter soft-deleted rows.
  Use `Where<T>` or `All<T>` for soft-delete-aware filtering, or add an
  explicit RQL clause like `eq(deleted_at, null)`.
- `Select<T>(SQL, params)` and `SelectOne<T>(SQL, params)` accept
  literal SQL ‚Äî the framework cannot inject the auto-filter. Add the
  `WHERE deleted_at IS NULL` clause manually if needed.
- Cascade soft-delete (deleting a parent automatically soft-deletes
  dependent children) is **not implemented**. It depends on
  relationship metadata which is part of the Lazy Loading quick win
  (deferred to a future round).

## Interactions matrix

| Feature √ó Feature | Behaviour |
|---|---|
| Soft Delete √ó `foVersion` | Transparent. Soft-delete passes through UPDATE ‚Üí version check + increment apply as usual. Bulk RQL forms (`DeleteRQL`, `HardDeleteRQL`, `RestoreRQL`) bypass version check (existing bulk semantics). |
| Soft Delete √ó Audit | `MVCAuditUpdatedAt` / `MVCAuditUpdatedBy` fire on soft-delete and on Restore (both are UPDATE statements). |
| Soft Delete √ó Validation | `OnInputValidate` and `OnStorageValidate` do **not** fire on soft-delete or Restore. These are marker operations, not content changes. |
| Soft Delete √ó Hooks | `OnBeforeDelete` / `OnAfterDelete` fire on both soft and hard delete. `OnBeforeUpdate` / `OnAfterUpdate` are suppressed during soft-delete dispatch (the user called Delete, not Update). |
| Soft Delete √ó `[MVCDirtyTracking]` | Soft-delete changes one tracked field (the marker). Snapshot is rehashed normally on Update. `IsDirty` correctly reports the marker as the only dirty field during the transition. |
| `foRefresh` √ó `[MVCDirtyTracking]` | Refresh runs after Insert/Update execute. The dirty snapshot is rehashed AFTER refresh, so the new snapshot reflects DB-truth values (not stale in-memory pre-Insert defaults). |
| `foRefresh` √ó `foVersion` | Independent. Version field is never `foRefresh`. |

## What's deferred

- **Lazy loading** of relationships (`TMVCLazy<T>` / `TMVCLazyList<T>`,
  `[MVCBelongsTo]`, `[MVCHasMany]`). Designed but deferred to a
  future quick-wins round.
- **Composite primary keys**. Estimated 7-12 days of cross-cutting
  refactor; moved to 4.0-oxygen scope.
- **Cascade soft-delete**. Depends on lazy-loading metadata.
- **ZeosLib backend**. Designed for 4.0-oxygen as alternative to FireDAC.

```pascal
MVCDirtyTrackingAttribute = class(MVCActiveRecordCustomAttribute);

TMVCActiveRecordFieldOption = (
  foPrimaryKey, foAutoGenerated, foReadOnly, foVersion,
  foDoNotSelect, foDoNotInsert, foDoNotUpdate,
  foRefresh                                    // <-- new

```
